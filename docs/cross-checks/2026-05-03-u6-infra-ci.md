## Gaps Analysis

### GAP-001: First production run is operational verification, not local verification

**Requirement**: The actual cron fire, Pages deployment, and Telegram delivery must work in GitHub after secrets are configured.

**Status**: ⚠️ Partial at operational level. Local verification confirms workflow syntax, command wiring, strict MkDocs build, entrypoint override tests, and tracked symlink presence. The real hosted environment depends on repository settings and secrets.

**Impact**: Low for u6 stage gate. This is the normal final operational check for GitHub Actions / Pages.

**Proposed Action**: After commit/push, verify the first `workflow_dispatch` run in GitHub Actions, check Pages deployment URL, and confirm Telegram channel/operator chat delivery with real secrets.

**2026-05-05 observation**: GitHub Actions run history was reachable via
`gh run list --repo github.com/murphyGo/investo`. The latest `pages` run
(`25284140520`) built the MkDocs site successfully, then failed in
`actions/configure-pages@v5` with `Get Pages site failed` /
`HttpError: Not Found`, which indicates the repository Pages setting
still needs **Source: GitHub Actions** enablement before the first
deploy. The latest `daily-briefing` run (`25293136315`) failed in
`Run pipeline (python -m investo)`; public run metadata exposes only
exit code 1, while failed logs require repository admin access. The
next operator checks are therefore: enable Pages for GitHub Actions,
verify the five required GitHub Secrets, then re-run `daily-briefing`
via `workflow_dispatch`.

**2026-05-06 follow-up**: Added a `Preflight required secrets` step to
`daily-briefing.yml` before `python -m investo`. Missing or empty
required secrets now fail the workflow with explicit GitHub Actions
annotations naming the missing secret variables, without echoing any
secret values. This keeps `__main__._validate_env` as the runtime
boundary while making hosted-run diagnosis faster.

**2026-05-06 follow-up 2**: Expanded the preflight to also reject equal
Telegram briefing/operator chat IDs and malformed `SITE_URL_BASE`
before the pipeline starts. The checks deliberately report only the
failed invariant or variable name, not the configured secret values.

**2026-05-06 follow-up 3**: Moved the preflight implementation from
inline workflow Python to `scripts/check_daily_briefing_env.py` and
added unit tests for valid env, missing secrets, chat-ID equality,
malformed site URL, redaction, and workflow wiring. The workflow now
calls the tested script directly.

**2026-05-07 observation**: Read-only Pages API check
(`gh api repos/murphyGo/investo/pages --hostname github.com`) returned
`HTTP 404 Not Found`, confirming the Pages site is still not enabled or
not visible to the current GitHub credentials. The next operator action
remains **Settings -> Pages -> Build and deployment -> Source: GitHub
Actions** before expecting the `pages` workflow to deploy.

Resolved on 2026-05-04. The u5 follow-up fix retries operator-alert delivery failures and logs a warning only after final failure. u6 continues to own only the GitHub Actions fallback side by preserving exit code 1 and logs.
