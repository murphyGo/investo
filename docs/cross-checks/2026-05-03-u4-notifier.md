## Gaps Analysis

### GAP-001: FR-004 / FR-007 chat separation is completed across u4 + u5

**Requirement**: Public briefing messages and operator failure alerts must use separate chats, and failure alerts must never be posted to the public channel.

**Status**: ⚠️ Partial at u4 unit level only. u4 provides separate dispatcher classes, kwargs-only constructors, and tests proving each class sends to the constructor-provided `chat_id`. The final invariant that the two environment variables are unequal belongs to u5, because u4 receives only one ID per dispatcher; u5 verifies that invariant.

**Impact**: Low for u4 stage gate. This is an intentional unit boundary, not an implementation miss inside u4.

**Action Taken**: `TELEGRAM_BRIEFING_CHANNEL_ID != TELEGRAM_OPERATOR_CHAT_ID` and failure routing were verified in the u5 orchestrator cross-check. Do not add a u4 TECH-DEBT item.

### GAP-002: FR-007 retry/log marking and GitHub Actions default notification are outside u4

**Requirement**: If the operator alert itself fails, retry and mark the GitHub Actions log explicitly; GitHub Actions default failure notification should also be active.

**Status**: ⚠️ Partial at full-product level. u4 correctly returns `SendResult(ok=False)` for alert send failures without raising. Retry policy, log marking, and workflow notification behavior are orchestration/CI concerns.

**Impact**: Low for u4 stage gate. u4 exposes the failure signal needed by downstream units.

**Action Taken**: Retry/log behavior was verified and closed in the u5 follow-up fix on 2026-05-04; GitHub Actions failure signaling is verified in the u6 report. Do not add a u4 TECH-DEBT item.
