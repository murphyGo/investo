# Code Generation Plan: `u19 briefing-visual-assets`

**Date**: 2026-05-07
**Unit**: u19 briefing-visual-assets
**Stage**: Code Generation

---

## Goal

Generate deterministic, data-derived visual cards for each daily market briefing so readers can understand market state, data confidence, and personal relevance before reading the full 7-section narrative.

The unit intentionally does **not** scrape and republish third-party news photos, chart screenshots, or unverified logos. Investo archives are public and permanent, so image assets must be generated from collected data or from explicitly licensed project-owned assets.

---

## User Findings

Five sub-agent reviews on 2026-05-07 converged on the same product shape:

- The highest-value first visual is a **data confidence card**: segment, coverage status, item count, source count, and missing categories.
- The second visual is a **market snapshot card**: the day's conclusion, main driver, caution, and data state.
- Price visuals are useful when they are based on stable raw source metadata:
  - `yfinance-price`: latest OHLCV, previous close, and percent change for configured US tickers.
  - `coingecko-price`: current price, 24h percent change, volume, market cap, 24h high/low for BTC/ETH/SOL.
- Watchlist visuals are useful as relevance cards, but must say "matched collected items" rather than inventing investment impact.
- Long time-series charts, domestic price charts, sector heatmaps, and scraped article photos are out of v1 scope.

---

## External Image Policy

Default policy: `scrape_external_images = false`.

Allowed:

- Images generated by Investo from collected structured data.
- Project-owned static assets.
- Explicitly licensed assets with a checked-in manifest containing source URL, license, attribution, author, fetched date, and allowed use.

Disallowed in v1:

- News article thumbnails or RSS `media:content` downloads.
- External chart screenshots from broker, finance, or charting websites.
- Logo scraping or automatic third-party brand image reuse.
- LLM/image-generation API calls during the scheduled briefing run.

Rationale:

- The U.S. Copyright Office describes fair use as a case-by-case four-factor analysis, not a blanket permission for public republishing.
- Google Search Help recommends checking the exact image license and reuse terms before reuse, even when an image search filter shows license information.
- Creative Commons assets still require license-specific conditions such as attribution.

References:

- <https://copyright.gov/fair-use/>
- <https://www.copyright.gov/help/faq/faq-fairuse.html>
- <https://support.google.com/websearch/answer/29508>
- <https://creativecommons.org/licenses/by/2.5/>

## Definition of Done

- [ ] Visual assets are generated deterministically without network access.
- [ ] Segmented briefings can include visual assets for domestic-equity, us-equity, and crypto.
- [ ] At least one data confidence card is generated from `SegmentCoverage`.
- [ ] At least one market snapshot card is generated from the cleaned summary header.
- [ ] US equity and crypto price cards are generated when source metadata is available, and omitted gracefully when unavailable.
- [ ] Watchlist relevance cards show matched item counts/titles only; they do not infer investment impact.
- [ ] External image scraping is disabled by default and guarded by an explicit policy test.
- [ ] Generated images are non-empty, non-blank, bounded in dimensions, and below a size cap.
- [ ] Markdown image links are relative, MkDocs-safe, and do not create broken public pages.
- [ ] Telegram summary behavior remains within the existing 4096 UTF-16 unit limit.
- [ ] Image generation failure policy is explicit and visible in run diagnostics.

## Recommended Architecture

### New Module

Create a visual asset module with pure data-to-asset boundaries:

- `src/investo/visuals/paths.py`
  - `visual_asset_dir(target_date, segment)`
  - `visual_asset_path(target_date, segment, name, extension=".svg" | ".png")`
- `src/investo/visuals/cards.py`
  - data classes for card inputs
  - extraction from `NormalizedItem`, `SegmentCoverage`, and `WatchlistImpact`
- `src/investo/visuals/render.py`
  - deterministic SVG or PNG renderer
  - text wrapping and truncation helpers
- `src/investo/visuals/policy.py`
  - external image policy constants and validation

Prefer SVG for v1 unless Telegram direct image attachment is implemented in the same unit. SVG keeps the repo footprint small, is text-testable, and works naturally in MkDocs. PNG can be added when Telegram media attachment becomes a hard requirement.

### Pipeline Integration

Add a publish-preparation stage after briefing generation and before archive publish:

1. collect source items
2. generate segmented briefings
3. prepare visual assets and insert relative markdown image links
4. publish markdown and assets together
5. notify Telegram with existing summary constraints

Do not put image generation in:

- `notifier`: Telegram should send summaries, not own site asset generation.
- `publisher.write_briefing`: publisher should remain verify/write/commit focused.
- LLM prompts: file paths and asset existence must not depend on LLM output.

### Asset Paths

Use markdown-adjacent asset directories:

```text
archive/{segment}/YYYY/MM/YYYY-MM-DD.md
archive/{segment}/YYYY/MM/YYYY-MM-DD.assets/summary-card.svg
archive/{segment}/YYYY/MM/YYYY-MM-DD.assets/data-confidence.svg
archive/{segment}/YYYY/MM/YYYY-MM-DD.assets/price-snapshot.svg
archive/{segment}/YYYY/MM/YYYY-MM-DD.assets/watchlist-relevance.svg
```

Markdown references should be relative:

```markdown
![데이터 신뢰도](2026-05-07.assets/data-confidence.svg)
```

This keeps MkDocs publication aligned with the existing `site_docs/archive -> ../archive` symlink contract.

## v1 Card Scope

### 1. Data Confidence Card

Inputs:

- `segment`
- `coverage.status`
- `coverage.item_count`
- `coverage.source_count`
- `coverage.missing_categories`
- `target_date`

Value:

- Prevents readers from over-trusting a data-limited market narrative.

Risk:

- Low. The required data is already deterministic.

### 2. Market Snapshot Card

Inputs:

- segment label
- target date
- cleaned conclusion
- cleaned main driver
- cleaned caution
- coverage status

Value:

- Gives readers a 3-second view of why the market matters today.

Risk:

- Medium. Header text is still derived from narrative output, so the existing summary quality contract must be reused before rendering.

### 3. Price Snapshot Card

Inputs:

- US equity: ticker, close, percent change, volume, high, low.
- Crypto: symbol, price, 24h percent change, 24h volume, market cap, high, low.

Value:

- Converts key numeric moves into a glanceable card.

Risk:

- Medium. `raw_metadata` is adapter-specific, so v1 should support only known source schemas and omit unknown rows.

### 4. Watchlist Relevance Card

Inputs:

- match count
- top matched terms
- source name
- matched item title
- optional URL

Value:

- Shows whether the briefing directly mentions the user's watched assets or themes.

Risk:

- Medium. Matching is string-based, so wording must avoid claiming causal portfolio impact.

### Out of Scope

- Scraped article photos.
- Scraped logo packs.
- Broker/charting website screenshots.
- Long historical line charts.
- Domestic-equity price charts before a domestic price adapter exists.
- Sector heatmaps before a stable sector taxonomy exists.

---

## Failure Policy

Recommended v1 policy:

- If visual generation fails before markdown links are inserted, continue publishing markdown and notify diagnostics with `visual_assets=failed`.
- If markdown already references visual assets but an asset file is missing or invalid, block publish to avoid broken public pages.
- Always include visual asset status in GitHub Step Summary:
  - generated asset count
  - paths
  - dimensions or SVG root size
  - byte size
  - validation status
  - failure reason when applicable

## Test Plan

### Unit Tests

- Path contract:
  - `visual_asset_dir(target_date, segment)` returns the markdown-adjacent `.assets` directory.
  - paths are repo-relative and segment/date-stable.
- Card input extraction:
  - `SegmentCoverage` maps to status/count/missing-category text.
  - known `yfinance-price` and `coingecko-price` metadata map to price rows.
  - unknown or missing metadata is omitted without raising.
  - `WatchlistImpact` renders match counts without impact claims.
- Rendering:
  - SVG/PNG has fixed dimensions.
  - output is non-empty and non-blank.
  - long Korean, long English ticker lists, markdown artifacts, and emoji do not overflow text boxes.
  - output excludes secret-like values by reusing leak-guard checks.
- Policy:
  - external image scraping is disabled by default.
  - assets without an allowed license manifest are rejected if external assets are ever enabled.

### Integration Tests

- Segmented pipeline happy path creates markdown plus asset files.
- Git staging includes markdown and generated assets in one commit.
- Missing generated asset blocks publish when markdown references it.
- Telegram summary remains under the 4096 UTF-16 unit limit.
